# CLAUDE.md

Claude Code 작업 지침입니다. 이 저장소(**이건영 GitHub 프로필**, `pachir1su/pachir1su`)에서
작업할 때는 아래 규칙을 우선 적용합니다.

## 프로젝트

- GitHub 프로필 페이지에 노출되는 **프로필 README 저장소**입니다. `README.md`가 곧 프로필 화면입니다.
- 페이퍼톤(크림 `#fffdf6` + 레드 `#c63b3b`) 테마를 기본으로 하며, 색·톤은 이 팔레트에 맞춥니다.
- `github-stats.svg`는 **`generate_stats.py`가 생성**합니다. 이 SVG는 직접 수정하지 말고
  스크립트를 고친 뒤 재생성합니다.
- `github-metrics.svg`(lowlighter/metrics), snake 애니메이션(Platane/snk, `output` 브랜치),
  streak·activity graph 등은 **워크플로/외부 서비스가 생성**합니다. 결과물을 직접 손대지 않습니다.
- `README.md`의 문구·레이아웃은 **수작업으로 관리**합니다.

## 브랜치 네이밍 규칙

브랜치 생성 시 반드시 아래 형식을 따를 것:

```
claude/kebab-description-N
```

| 항목 | 규칙 |
|---|---|
| `claude/` | 고정 prefix, 절대 생략 금지 |
| `kebab-description` | 2~5단어, 소문자, 하이픈 구분 |
| `-N` | 기존 `claude/` 브랜치 최대 번호 + 1 |

1. `git branch -a | grep 'claude/'` 로 기존 번호 확인
2. 최대 번호 + 1 을 N으로 사용 (없으면 N=1)
3. `git checkout -b claude/your-slug-N`
4. 랜덤 문자열(예: `claude/beautiful-hopper-FvyrU`) 절대 사용 금지

> 별도로 개발 브랜치가 지정된 세션에서는 지정 브랜치를 우선합니다.

## 작업 방식

- **구현 전 설계 방향을 먼저 확인**하고, 파일 수정 전 기존 코드·문구를 파악한 뒤 진행합니다.
- 특히 `README.md` 문구·디자인은 **임의로 바꾸지 말고, 진행 전 사용자에게 먼저 확인**합니다.
- 한 작업 안에서 커밋은 되도록 잘게, 많이 쪼갭니다(데이터 → 스크립트 → 스타일 → 문서 등 단위별로).
- 큰 변경은 단계별로 나눠서 진행합니다.
- QA·검증할 항목이 있으면 PR 본문에 정리해 넣습니다(무엇을 어떻게 확인했는지, 남은 확인 사항 포함).
- 사용자가 직접 해야 할 일(워크플로 수동 실행, 시크릿 발급, 실기기·브라우저 확인 등)이 있으면
  하나하나 구체적으로 알려줍니다.
- 궁금하거나 불확실한 점은 임의로 결정하지 말고 항상 먼저 물어봅니다.
- **PR을 주기적으로(예: 1시간마다) 자동 재검증하거나 셀프 체크인을 예약하지 않습니다.**
  사용자가 명시적으로 요청할 때만 합니다.

## 이슈와 PR

- **PR 댓글의 "ㅇㅇ"는 "곧 머지하겠다"는 뜻입니다.** 추가 작업을 시작하라는 지시가 아니므로,
  그걸 근거로 새 작업을 벌이지 않습니다. 다음 작업은 사용자가 명시적으로 요청할 때만 시작합니다.
- 닫힌 이슈는 사용자가 명시적으로 요청하지 않는 한 읽거나 근거로 삼지 않습니다.
- 이슈를 확인할 때는 본문뿐 아니라 댓글도 확인합니다.
- 커밋은 이슈나 작업 단위로 작게 쪼갭니다.
- PR 본문에는 관련 이슈 번호가 있으면 명확히 연결합니다.
- 푸시 후 해당 브랜치에 열린 PR이 없으면 **draft PR**을 생성합니다.
- 불확실한 요구사항은 구현 전에 사용자에게 질문합니다.

## 코드 변경

- 기본 언어는 **Python**(별도 지시 없을 시)입니다. 기존 구조와 네이밍 스타일을 우선 따릅니다.
- 변수·함수명은 `camelCase`, 코드 블록마다 역할 주석을 답니다. 실행 시 예외 처리를 잊지 않습니다.
- 성능·가독성을 함께 챙기고, 불필요한 리팩터링과 범위 밖 변경은 피합니다.
- 비밀값, 실제 토큰, 개인 설정 파일을 출력하거나 커밋하지 않습니다.
- 생성물(`github-stats.svg`, `github-metrics.svg`, snake) 자체를 직접 수정하지 않습니다.

## 시크릿·토큰

- 워크플로는 저장소 시크릿(`METRICS_TOKEN` 등)을 사용합니다. **값이 필요하거나 바꿔야 하면
  직접 만지지 말고, 무엇을 어떻게 설정할지 사용자에게 알려줍니다.**
- 토큰·계정 실제 값을 문서·코드·PR 본문에 노출하지 않습니다.

## 커밋 메시지 규칙

```
<type>: <short description>
```

| type | 용도 |
|---|---|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `refactor` | 리팩토링 |
| `docs` | 문서 수정 |
| `chore` | 설정, 의존성 등 |

## 검증

- **스크립트 변경**: `python -m py_compile generate_stats.py`로 구문을 확인합니다.
  `generate_stats.py`는 `GITHUB_TOKEN`이 있어야 실제 실행되므로, 실행 검증이 필요하면
  토큰 설정 방법을 사용자에게 안내합니다.
- **SVG·데이터 변경**: 생성 결과를 XML로 파싱해 유효성을 점검하고, 가능하면 렌더를 확인합니다.
- **README·문서만 변경한 경우**에도 링크와 뱃지·이미지 URL을 직접 확인합니다.
- PowerShell에서 한글 문서를 읽을 때는 `Get-Content -Encoding UTF8`을 사용합니다.
